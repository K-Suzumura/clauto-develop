## 作成のポイント

### 社外向け

1. **ユーザー視点で記載**
   - 技術的な詳細より、ユーザーへのメリットを強調
   - 「〇〇ができるようになりました」形式

2. **破壊的変更は目立たせる**
   - 影響範囲を明確に
   - 移行手順を具体的に

3. **既知の問題は正直に**
   - 隠さず記載
   - 回避策があれば提示

### 社内向け

1. **技術詳細を記載**
   - 変更ファイル、設定変更
   - DB マイグレーション情報

2. **運用情報を充実**
   - デプロイ手順
   - ロールバック手順
   - 監視ポイント

3. **連絡先を明確に**
   - 問題発生時の連絡先
   - 担当者の明示

## バージョニング

### セマンティックバージョニング

```
MAJOR.MINOR.PATCH

MAJOR: 破壊的変更
MINOR: 後方互換性のある新機能
PATCH: 後方互換性のあるバグ修正
```

### 例

| 変更内容 | バージョン変更 |
|---------|--------------|
| 新しい API エンドポイント追加 | MINOR |
| 既存 API のレスポンス形式変更 | MAJOR |
| バグ修正 | PATCH |
| セキュリティ修正 | PATCH |
